Contexto del proyecto

Problema, solución y resultado.

El problema

El restaurante gestionaba los pedidos en papel: los garzones escribían comandas a mano, la cocina recibía notas físicas y el cierre de caja era manual. Esto generaba errores frecuentes, pérdida de información y tiempos muertos innecesarios entre cada turno.

La solución

Un sistema web con tres módulos en tiempo real: el garzón toma pedidos desde su panel, la cocina los recibe automáticamente cada 4 segundos y puede actualizar su estado, y el administrador accede a KPIs, historial y gestión completa del menú.

El resultado

  • Eliminación total de comandas en papel
  • Tiempo real entre garzón y cocina sin fricción
  • Historial automático de cada pedido cerrado
  • Control de menú desde el panel admin
  • Dashboard de KPIs para decisiones rápidas

Qué demuestra

  • Arquitectura de sistemas con múltiples roles
  • Backend funcional con PHP y MySQL
  • API REST con 13 endpoints bien definidos
  • UX funcional para usuarios no técnicos
  • Pensamiento de producto end-to-end
Funcionalidades

Qué hace cada módulo.

01

Panel Garzón

Vista de mesas con estado en tiempo real. Permite abrir mesa, tomar pedido con detalle por producto, cantidad y nota, y finalizar mesa con un clic.

garzon/index.html
02

Panel Cocina

Muestra tarjetas de pedidos pendientes con polling automático cada 4 segundos. El cocinero puede cambiar el estado a "Preparando" o marcar como "Listo".

cocina/index.html
03

Panel Admin — KPIs e Historial

Consulta los últimos 50 pedidos cerrados con métricas clave. Permite visualizar ventas, tiempos y el comportamiento del restaurante en el tiempo.

admin/index.html
04

Panel Admin — Gestión de Menú

Alta, baja y toggle de disponibilidad de productos. Permite agregar nuevos platos, eliminarlos y marcarlos como agotados sin necesidad de tocar código.

admin/index.html
05

Cierre y Archivado Automático

Al marcar un pedido como listo, el sistema lo archiva automáticamente en historial_pedidos, libera la mesa y limpia los registros activos de la BD.

historial.php

Metodología

Cómo se construyó.

Fase 01

Análisis del flujo real

Se estudió cómo funciona un restaurante: qué necesita el garzón, qué necesita la cocina y qué datos requiere el administrador para tomar decisiones.

Fase 02

Diseño de base de datos

Se modeló la BD con tablas para mesas, pedidos, detalle de pedidos, productos e historial, asegurando integridad relacional y facilidad de consulta.

Fase 03

API y backend PHP

13 endpoints para manejar todos los flujos: guardar pedido, cambiar estado, cerrar mesa, gestionar menú y consultar historial con KPIs.

Fase 04

Interfaces y experiencia

Cada panel fue diseñado para su usuario: garzón con vista de mesas, cocina con tarjetas en tiempo real y admin con dashboard completo.
